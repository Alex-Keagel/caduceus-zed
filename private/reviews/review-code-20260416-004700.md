## Findings

### 🔴 Critical
1. `[crates/agent/src/thread.rs:1455-1505]` Ring-0 is not actually read-only. `edit_file`, `save_file`, and `create_directory` are allowlisted for `plan`, `research`, `architect`, and `review`, with no runtime path/extension filter. Research/architect/review prompts say “Do NOT modify any files,” but the runtime still allows writes. **Fix:** remove generic write tools from the Ring-0 allowlist and add path-aware validation at execution time.
2. `[crates/agent/src/tools/caduceus_mode_request_tool.rs:85-131, crates/agent/src/agent.rs:1562-1565, 1716-1733]` Mode escalation via `caduceus_mode_request` is broken. The tool returns “Mode changed” but never mutates `CaduceusSessionModes` or `Thread.caduceus_mode`. The UI mode selector path exists; the tool is not wired into it. **Fix:** either wire the tool into `set_mode(...)`, or stop claiming the mode changed.
3. `[crates/agent/src/tools/caduceus_project_wiki_tool.rs:88-109, 193-209]` Arbitrary file read/write via wiki path traversal. `page_path()` blindly joins user-controlled page names; `auto_populate()` also uses unsanitized repo names in `repos/{name}`. `../../` and absolute paths escape `.caduceus/wiki`. **Fix:** reject non-normal path components and return `Result<PathBuf, String>` from `page_path()`.
4. `[crates/agent/src/thread.rs:3304-3317]` Repository-controlled instructions are injected directly into the live system prompt. A malicious `AGENTS.md`/`CADUCEUS.md` can issue hostile system-level instructions. Truncation to 2000 chars does not make this safe. **Fix:** do not inject repo-authored text as system-role instructions; demote it to untrusted context or lower-priority user/context content.

### 🟡 Important
1. `[crates/agent/src/tools/caduceus_project_tool.rs:145-155]` `project.json` mutations are unlocked read-modify-write. Concurrent tool/subagent calls can lose updates or leave partially written state. `caduceus_file_lock` exists but is unused here. **Fix:** lock the file for the whole mutation and use atomic temp-file + rename writes.
2. `[crates/agent/src/tools/caduceus_project_wiki_tool.rs:101-108]` Wiki writes are also unlocked, so concurrent page writes can race and corrupt state. **Fix:** use `acquire_file_lock()` around wiki writes.
3. `[crates/agent/src/thread.rs:1513-1641]` Compaction telemetry is wrong. `fill_pct` ignores system-prompt overhead and mention/image payloads, and `tokens freed = total_tokens - estimate_tokens(summary)` is not the number of freed tokens. **Fix:** count mention/image/system-prompt overhead and compute freed tokens from removed messages minus inserted summary.
4. `[crates/agent/src/thread.rs:1795-1798]` The engine-sharing fallback silently creates a per-thread engine when the shared project engine is missing. That masks lifecycle bugs and can split index state. **Fix:** log loudly or skip engine-dependent tool registration instead of silently forking state.

### 🟢 Suggestions
1. Add targeted tests for: mode-request actually changing modes; Ring-0 code-write rejection; wiki traversal rejection; repo-name traversal rejection; concurrent `project.json` writes; compaction token estimation with mentions/images/system prompt.
2. Move mode-specific permission policy out of prompt prose and into a single input-aware enforcement layer.

## Recommended Actions
1. Fix mode escalation wiring before shipping; right now the main privilege-escalation path is false advertising.
2. Block wiki traversal and Ring-0 generic writes immediately.
3. Demote repo-authored instructions from system prompt authority.
4. Add locking + atomic writes for `.caduceus/project.json` and wiki pages.
